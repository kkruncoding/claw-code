## 仓库本质上

面向开发者的 AI 命令行代理产品（CLI agent）

## 核心价值

  1. 让用户在终端里和 Claude 交互
  2. 让 Claude 能调用本地工具

    - 读文件
    - 写文件
    - grep / glob
    - bash
    - web fetch / search
    - agent / skill / notebook 等
  3. 把这些能力做成可持续会话

    - 会话保存 / resume
    - 权限控制
    - 配置加载
    - MCP 服务接入
    - slash commands
  4. 兼顾产品体验

    - REPL 交互
    - 流式输出
    - markdown 渲染
    - 子命令
    - JSON 输出
    - 登录 / provider / stats / doctor

---
  产品角度下，它的核心用户流程

  你可以先只看用户会经历什么：

  1. 启动产品

  入口：
  - rust/crates/rusty-claude-cli/src/main.rs

  用户可能这样用：
  claw
  claw prompt "explain this repo"
  claw login
  claw status

  这对应产品层的“启动面”和“交互入口”。

---
  2. 进入会话

  产品开始建立运行上下文：
  - 当前目录
  - 配置文件
  - 模型
  - 权限策略
  - session 恢复
  - memory / CLAUDE.md
  - MCP / tools / skills

  这部分主要是：
  - runtime

---
  3. 用户发出请求

  比如：
  - 普通自然语言
  - slash command：/status、/doctor
  - 子命令：prompt、login、agents

  这时产品需要判断：
  - 这是聊天请求？
  - 命令请求？
  - 工具调用请求？
  - 是否允许执行？

  这部分会涉及：
  - commands
  - runtime
  - tools

---
  4. 模型生成响应并调用工具

  如果模型要读文件/执行命令：
  - tool schema 由 tools 提供
  - 执行过程由 runtime 编排
  - API 通信由 api 提供

  所以从产品能力看，最核心闭环是：

  CLI 壳层 → Runtime 编排 → API 调模型 → Tools 执行 → 再回到 CLI 展示

---
  5. 展示结果并持久化

  产品要负责：
  - 流式输出
  - markdown 渲染
  - tool call 展示
  - session 保存
  - telemetry / usage

  这里主要还是：
  - rusty-claude-cli
  - runtime
  - telemetry

---
  如果按“产品模块”拆，最核心是这 5 块

  1. rusty-claude-cli

  产品外壳 / 用户交互层

  你可以把它理解成：
  - CLI 启动器
  - REPL
  - 参数解析
  - 输出渲染
  - 用户能直接感知的命令入口

  它回答的是：

  ▎ “这个产品怎么被启动、怎么与用户交互、怎么展示结果？”

---
  2. runtime

  产品大脑 / 调度中枢

  这是最核心模块之一。

  它负责：
  - 会话状态
  - 配置加载
  - 权限判定
  - 系统提示词组装
  - MCP 生命周期
  - tool 调度上下文
  - session persistence / resume

  它回答的是：

  ▎ “产品在一次会话里，到底如何运转？”

  如果你只深入一个核心 crate，优先看 runtime。

---
  3. tools

  产品能力层

  它定义并执行产品能做的“动作”：
  - Bash
  - Read / Write / Edit
  - Grep / Glob
  - WebSearch / WebFetch
  - Agent
  - Skill
  - NotebookEdit

  它回答的是：

  ▎ “这个 AI 产品到底能替用户做什么事？”

  从产品角度，tools 就是能力目录。

---
  4. api

  模型/Provider 接入层

  负责：
  - Anthropic/OpenAI/xAI/DashScope 等 provider 接入
  - SSE streaming
  - 请求响应结构
  - 认证

  它回答的是：

  ▎ “这个产品怎么和大模型服务通信？”

---
  5. commands

  产品命令面 / 交互协议层

  负责 slash commands：
  - 注册
  - 解析
  - 帮助文本渲染

  它回答的是：

  ▎ “用户输入 /xxx 时，系统如何识别和处理？”

  你刚选中的
  - omc_compatibility_note_for_unknown_slash_command

  就属于这类“命令体验兼容性”的产品细节。

---
  针对你背景，我建议你的阅读方式

  你熟 C++，所以你读 Rust 时不要按“语法学习”驱动，而要按下面方式：

  读法 1：先问“职责”

  先看模块是干什么的，不要先纠结生命周期。

  读法 2：抓入口函数

  比如从 main.rs 往下跟调用。

  读法 3：重点看数据结构

  Rust 项目里，很多设计意图体现在：
  - struct
  - enum
  - trait

  读法 4：看到 Result / Option 不要怕

  把它当：
  - 显式错误流
  - 显式空值流

  读法 5：先忽略复杂泛型

  先看“这个函数在产品流程里干什么”。
